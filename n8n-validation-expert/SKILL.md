# n8n 验证专家

新版 MCP 的验证主语是“整段工作流代码”，不是孤立节点配置。

所以正确心智模型是：

```text
写代码 -> validate_workflow -> 读报错 -> 修代码 -> 再验证
```

通常要来回 2 到 3 轮，这是正常的。

---

## 先看什么

### 1. 先分级

- **Errors**：必须修，不然工作流创建或更新过不去
- **Warnings**：能运行，但大概率有风险
- **Suggestions**：不是阻塞项，可以排后面

### 2. 再分来源

常见来源只有几类：

- 代码语法不对
- 节点类型不对
- 参数结构不对
- 节点连接不对
- 表达式不对

不要一上来就大改整份代码，先判断报错属于哪一类。

---

## 推荐验证循环

### 标准流程

```javascript
const result = validate_workflow({ code });
```

然后按这个顺序处理：

1. 先修语法和结构性错误
2. 再修节点类型和参数类型
3. 再修连接和引用
4. 最后处理警告和建议

### 为什么这个顺序更稳

因为前面的错误经常会“带出”后面的假问题。

例如：

- 代码块括号没闭合
- 节点 ID 写错
- 某段参数结构层级写错

这些问题不修，后面的报错信息会越来越乱。

---

## 高频错误

### 1. 语法错误

表现：

- 代码根本没法解析
- 一串看起来不相关的错误一起冒出来

优先检查：

- 括号
- 逗号
- 字符串引号
- 对象和数组层级

### 2. 节点类型错误

表现：

- 节点类型不存在
- 节点名像是猜出来的

修法：

```text
search_nodes -> get_node_types -> 回填正确节点 ID
```

### 3. 参数类型或结构错误

表现：

- 字段名不对
- 某个字段应该是对象，你给成了字符串
- 某个字段只在特定操作下存在

修法：

```text
重新获取当前操作的类型定义 -> 对照修 node(...)
```

### 4. 连接错误

表现：

- 节点孤立
- 连接到错误分支
- AI 节点连接类型不对

修法：

- 先确认工作流主路径是否完整
- 再确认特殊连接类型是否正确

### 5. 表达式错误

表现：

- `{{ }}` 漏写
- 引用了不存在的数据路径
- 字段名含中文或特殊字符时写法不对

修法：

- 回到 `n8n-expression-syntax`
- 优先确认当前节点实际拿到的数据结构

---

## 不再推荐的旧思路

- 不再依赖旧版的节点级验证接口
- 不再区分旧版 profile
- 不再依赖旧版的自动修复接口
- 不再建议靠局部补丁修工作流碎片

新版路径就是：完整代码验证，完整代码修正。

---

## 排错优先级

### 必须立刻处理

- 语法错误
- 节点类型不存在
- 必填字段缺失
- 连接断裂

### 可以第二轮处理

- 警告
- 建议
- 最佳实践类提醒
