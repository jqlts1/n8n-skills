# 验证指南

新版 MCP 的验证对象是完整工作流代码：

```javascript
validate_workflow({ code })
```

不再是旧版那种围绕单个节点做多种 profile 验证。

---

## 验证主流程

```text
写 SDK 代码 -> validate_workflow -> 读错误 -> 修代码 -> 再验证
```

通常需要两到三轮，这是正常的。

---

## 什么时候必须验证

- 创建工作流前
- 更新工作流前
- 对主链路做了结构性修改后
- 换了关键节点或关键操作后

下面这些情况可以暂时不验证：

- 只是搜索节点
- 只是读类型定义
- 只是讨论模式

---

## 常见错误类型

### 1. 代码语法错误

表现：

- 解析失败
- 第一条错误就让整段代码失效

优先修：

- 括号
- 逗号
- 引号
- 对象层级

### 2. 节点类型错误

表现：

- 节点 ID 不存在
- 节点像是猜出来的

修法：

```text
search_nodes -> get_node_types -> 回填正确 nodeId
```

### 3. 参数结构错误

表现：

- 字段名不对
- 类型不对
- 某个字段在当前操作下不存在

修法：

```text
重新取当前操作的类型定义 -> 对照修 node(...)
```

### 4. 连接错误

表现：

- 节点链路断了
- 分支接错
- AI 连接类型不对

修法：

- 先看主执行路径
- 再看特殊连接

### 5. 表达式错误

表现：

- `{{ }}` 漏写
- 数据路径不对
- 当前节点取不到目标字段

修法：

- 先核对输入数据结构
- 再修表达式

---

## 推荐修复顺序

### 第一轮

- 修语法
- 修节点 ID

### 第二轮

- 修参数结构
- 修连接

### 第三轮

- 修表达式
- 清 warnings / suggestions

这个顺序的好处是：前面的错误经常会引发后面一串假问题。

---

## 遇到错误时配套用什么工具

### 参数看不懂

```javascript
get_node_types({nodeIds: [...]})
```

### SDK 语法不确定

```javascript
get_sdk_reference({section: "patterns"})
get_sdk_reference({section: "rules"})
get_sdk_reference({section: "functions"})
```

### 节点本身用法不确定

回到：

- `n8n-node-configuration`
- `n8n-node-pitfalls`

---

## 常见误区

### 误区 1：验证一次不过，就开始大改

先看最前面的错误。很多时候只要修第一处，后面一串都会消失。

### 误区 2：把验证当成创建后的补救

正确顺序是先验证，再创建 / 更新。

### 误区 3：参数错了还硬猜

参数不确定时不要猜，直接回 `get_node_types`。

### 误区 4：只看错误，不看 warnings

如果目标是先跑通，可以先修 errors；但准备上线时，warnings 不能长期堆着不管。

---

## 最佳实践

- 每次创建和更新前都验证
- 一次优先修一类问题，不要同时乱改
- 参数问题回类型定义，语法问题回 SDK 参考
- 保留验证循环心智，不期待一轮全过
