# 字段依赖指南

新版 MCP 下，不再通过单独的依赖查询工具理解字段关系。

应该把“字段依赖”理解为：某些字段是否出现、是否必填、值是否合法，取决于别的字段值。而这些规则现在主要从 `get_node_types` 返回的类型定义里读出来。

---

## 推荐做法

```text
search_nodes -> get_node_types(带 discriminators) -> 锁定 resource/operation -> 读字段关系
```

如果搜索结果已经告诉你某个节点支持不同资源、操作或模式，拿这些信息去请求更具体的类型，而不是只拿一个泛节点定义。

---

## 常见依赖类型

### 1. 操作切换型

最常见。

```text
resource / operation 变了
-> 可见字段变了
-> 必填字段也跟着变
```

典型例子：

- Slack 发消息和更新消息
- 数据库查询和插入
- Gmail 发送邮件和读取邮件

### 2. 方法切换型

尤其常见于 HTTP Request。

```text
method = GET
-> 通常没有 body

method = POST
-> 常常需要 sendBody 和 body 结构
```

### 3. 开关控制型

某个布尔值打开后，后面一整组字段才有意义。

典型例子：

- `sendBody`
- `sendQuery`
- `usePagination`
- `jsonParameters`

### 4. 认证方式型

认证方式不同，所需字段和凭证配置也不同。

不要把“无认证”的示例直接搬到“预定义凭证”场景。

---

## 读类型时重点看什么

### 1. 当前例子是不是已经锁定了操作

如果没有锁定到具体操作，读到的定义往往太宽。

### 2. 字段是“总存在”还是“条件存在”

如果字段只在特定条件下出现，就不要把它当作基础必填项教给用户。

### 3. 表达式位置是不是值字段

表达式通常应该出现在值位置，而不是结构字段名位置。

---

## 实战判断

### 遇到“为什么这里突然多出一组参数”

先看前面是不是切了操作、方法或认证方式。

### 遇到“为什么我照着别的例子抄却报缺字段”

通常不是节点错了，而是你抄的是另一个操作。

### 遇到“我不知道先补哪个字段”

先把当前操作下的最小可运行形状写出来，再跑 `validate_workflow`，让真实报错指路。

---

## 不再推荐的做法

- 不再依赖旧版的字段依赖查询接口
- 不再把“字段依赖”理解成独立工具返回的一张清单
- 不再把旧版 JSON 配置片段视为最终答案

现在的主路径始终是：类型定义 + 工作流代码验证。
